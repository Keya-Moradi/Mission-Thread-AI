# Decisions

Format: Date · Decision · Why · Alternatives considered.

## 2026-07-17 — Node 24.x pinned as target LTS

Why: Active LTS with the longest remaining support window as of this date (Active until Oct 2026, Maintenance until Apr 2028). Node 25 (odd majors never receive LTS status) is already end-of-life as of this writing and must not be used for development or builds.
Alternatives: Node 22 (Maintenance LTS already, shorter remaining active window).

## 2026-07-17 — Prisma schema centralized in packages/core

Why: the deterministic services (`getImpactedRequirements`, `calculateBudgetExposure`, etc.) and the future read-only MCP server both need direct DB access via `packages/core`; one schema avoids drift between two copies.
Alternatives: schema in `apps/web/prisma` (rejected — would force `packages/mcp-server` to depend on `apps/web`).

## 2026-07-17 — Three Prisma model merges approved for MVP

`TestResult` → `TestCase`, `SupplierUpdate` → `ProgramEvent`, `Approval` → `Decision`. Net model count 23 → 20.
Why: user requested junior-dev-level MVP complexity; each merged pair had no independent lifecycle need within MVP scope (no test run history, only one event type in MVP, approval and decision are always created 1:1 together). `AuditEvent` stays separate in all three cases as the append-only log entry.
Alternatives: build all 23 models exactly as listed in `SPEC.md` §6 (available later if a real need for the split emerges — e.g. a second event type would justify splitting `SupplierUpdate` back out of `ProgramEvent`).
Status: approved by user during Phase 0 plan review.

## 2026-07-18 — Prisma Client generator: `prisma-client-js` (not the newer `prisma-client` provider)

Why: Prisma 7 introduced a new `prisma-client` generator that emits raw TypeScript into a custom `output` path instead of compiled JS into `node_modules`. That raw-TS output uses `.js`-suffixed relative imports pointing at `.ts` files (standard TS `NodeNext` convention) — Turbopack (Next 16's dev/build bundler) could not resolve those specifiers when the package was loaded through `transpilePackages`, producing `Module not found` errors even inside Prisma's own generated files. Switching to the classic (now-deprecated-but-supported) `prisma-client-js` provider generates real compiled JS into `node_modules/@prisma/client`, which Next resolves through normal node_modules resolution with no special handling needed.
Consequence: `packages/core` imports `PrismaClient` from `"@prisma/client"` rather than a local generated path. Prisma 7 still required removing the schema's `datasource.url` in favor of `prisma.config.ts` + a driver adapter (`@prisma/adapter-pg`) regardless of generator choice — that part is unrelated to this decision.
Alternatives: keep `prisma-client` with custom `output` (rejected — broke Turbopack resolution, verified by reproducing the failure with `npm run dev`); patch Turbopack's `resolveExtensions`/`resolveAlias` (rejected — those options don't remap an explicit `.js` specifier to an existing `.ts` file, verified against Next's own docs).

## 2026-07-18 — `next-auth` pinned to the v5 beta channel

Why: Auth.js's own current documentation recommends `next-auth` v5 for Next.js App Router support; v4 predates the App Router pattern used here (Credentials provider, `auth()` in server components, Server Action sign-in). As of this date, `next-auth@latest` on npm still resolves to the v4 line (`4.24.14`); v5 (`5.0.0-beta.31`) is only available under the `beta` dist-tag.
Consequence: this is a pre-1.0 dependency and could introduce breaking changes on a version bump. Documented as a known risk in `README.md`.
Alternatives: use v4 with a Pages-Router-style compatibility shim (rejected — not the officially documented App Router path, more fragile long-term).

## 2026-07-18 — No Next.js middleware/proxy for authentication in Phase 1

Why: the revised Phase 1 instructions require not importing Prisma or `node:crypto` into middleware, and verifying Auth.js's current split-config pattern before adding middleware at all. Auth checks are done with `auth()` directly inside server layouts, pages, and the login Server Action instead.
Consequence: there is no session-refresh-on-every-request behavior that middleware would normally provide; acceptable for Phase 1's scope (no mutations exist yet). Revisit if/when a route genuinely needs edge-level gating.
Alternatives: add `proxy.ts` (Next 16's replacement name for `middleware.ts`) now (rejected — adds Edge-runtime constraints and a second auth config surface before there's a concrete need).

## 2026-07-18 — `apps/web/.env` is a symlink to the repo-root `.env`

Why: `SPEC.md` §17 requires `.env.example`/`.env.test.example` at the repo root, but Next.js only auto-loads `.env`/`.env.local` from the Next app's own directory (`apps/web/`), not the monorepo root. A symlink keeps one canonical `.env` file instead of duplicating secrets.
Consequence: a fresh clone must run `ln -s ../../.env apps/web/.env` once after copying `.env.example` — documented in `README.md`. The symlink itself isn't committed (it's local machine state, and `.env` is gitignored).
Alternatives: duplicate `.env` into `apps/web/` (rejected — two copies of the same secrets drift silently); configure Next to read a custom env path (rejected — not a standard, documented Next.js option).

## 2026-07-18 — Accepted 3 moderate `npm audit` advisories rather than force-fixing

Why: `npm audit fix --force` would downgrade `prisma` to `6.19.3` and `next` to a `9.x` canary to resolve advisories in nested/optional dev-tooling dependencies (`@prisma/dev`'s use of an old `@hono/node-server`, and a `postcss` copy bundled inside Next itself). Both advisories are in code paths this app doesn't exercise (an optional Prisma dev-studio dependency; Next's internal CSS stringification). Downgrading two major framework versions is a materially worse trade than the advisories themselves.
Consequence: `npm audit` will report 3 moderate findings until upstream ships non-breaking patches. Documented in `README.md` Limitations.
Alternatives: force-fix now (rejected — breaking downgrade); ignore silently (rejected — documented instead, per the project's own rule to record risks honestly).

## 2026-07-18 — Destructive-operation authorization policy

- Every destructive database invocation requires fresh human authorization.
- Approval applies only to the named command and target database.
- Previous approval must not be reused for a later destructive invocation.
- Third-party safety controls must never be bypassed or fabricated.
- The application's own allowlists and guards (`packages/core/src/db-safety.ts`) remain required even when external tooling also provides safety controls.

Why: destructive database commands (schema reset, full-table clear-and-reseed) are irreversible against whatever they target. Authorization has to be re-established per invocation, not carried forward from an earlier one, or "approval" stops meaning anything.
How to apply: this governs every destructive command in this repository going forward, regardless of which tool or terminal runs it.

## 2026-07-18 (correction pass) — Hardened `verifyPassword` against a malformed-hash authentication bypass

Confirmed as a real, reproducible defect: `Buffer.from(str, "hex")` silently stops decoding at the first invalid character instead of throwing, so a corrupted stored hash (bad migration, manual edit, future bug) with non-hex salt/hash fields could decode to two empty buffers. `scryptSync` happily derives a 0-byte key from a 0-byte salt, and `timingSafeEqual` on two empty buffers returns `true` — authenticating **any** password against that row. Reproduced directly in Node before fixing: `verifyPassword("literally anything", "scrypt:16384:8:1:zz:zz")` returned `true`.
Fix: `packages/core/src/auth/password.ts` now fully validates every field (exact marker, field count, integer-only cost params in bounds, power-of-two N, a combined `128*N*r` memory ceiling, even-length hex within a plausible byte-length range) before any bytes are ever passed to scrypt, and wraps the scrypt call itself in try/catch so any unanticipated error fails closed instead of throwing out of an auth check.
Async decision: switched `hashPassword`/`verifyPassword` from `scryptSync`/`timingSafeEqual`-only to `crypto.scrypt` (promisified) so a slow verify (intentionally CPU/memory-heavy by design) never blocks Node's single event loop thread during a login attempt. This was straightforward here since the only caller (`apps/web/src/auth.ts`'s `authorize()`) was already async.
Alternatives: keep the length checks tied exactly to the current `SALT_LENGTH`/`KEY_LENGTH` constants (rejected — would break verification of any hash created under different future parameters; used a safe range instead, matching the format's own stated goal of tolerating parameter changes).

## 2026-07-18 (correction pass) — Shared destructive-database-operation guard

**Superseded by the 2026-07-18 second correction pass ("Exact (host, port, database) target tuples...", "Destructive commands renamed...") and the 2026-07-18 third correction pass ("Exact GitHub Actions target requirement", "Target-specific destructive seed commands") below.** The independent host/database-name allowlists, the permanently-enabled example-file flag, the `db:seed`/`db:seed:destructive` command names, and the `CI=true` check described in this entry no longer reflect current behavior — kept here as the historical record of the original fix, not as a description of the present design.

`packages/core/src/db-safety.ts`'s `isTestDatabaseName` used `.includes("test")`, a plain substring match that would also match `contest_prod`, `latest`, `attestation`, and `testament`. It also had no allowlist, no production/host checks, and — critically — the seed script's `clearExistingData()` (which wipes every table) had **no guard of any kind**, running unconditionally on every `npm run db:seed`. Separately, one error path in the old `reset-test-db.ts` interpolated the raw `TEST_DATABASE_URL` (including credentials) directly into a `console.error` call.
Fix: replaced the substring check with a token-boundary regex (`test` must be delimited by the string start/end or `_`/`-`) plus an explicit `APPROVED_TEST_DATABASE_NAMES`/`APPROVED_DEV_DATABASE_NAMES` allowlist, and added one shared `checkDestructiveOperationAllowed()` guard used by both `reset-test-db.ts` and `seed.ts`'s clear step. It fails closed on: `NODE_ENV=production`, a missing/malformed URL, a host outside `APPROVED_LOCAL_HOSTS` (localhost/127.0.0.1/::1/postgres — no remote-host opt-in exists, since nothing in this project legitimately needs one yet), a database name outside the operation's specific allowlist, or a missing `ALLOW_DESTRUCTIVE_DATABASE_OPERATION=true` opt-in (exact-match only — "1"/"yes"/"TRUE" do not count). All returned messages are pre-sanitized to host/port/database only; the raw connection string is never logged.
Consequence: `npm run db:seed` now requires `ALLOW_DESTRUCTIVE_DATABASE_OPERATION=true` in the environment — added to `.env.example`/`.env.test.example` (and CI) so local/CI workflows are unaffected, but a misconfigured `DATABASE_URL` without that flag (or pointed at an unrecognized host/name) now fails safely instead of silently wiping data.
Alternatives: keep the two scripts' checks independent (rejected — that's exactly how the seed script ended up with no check at all); let the flag alone gate remote hosts too (rejected — kept the host allowlist unconditional for a stronger guarantee, since no real use case needs an override today).

## 2026-07-18 (correction pass) — Deterministic IDs for all seeded rows; distinct password hash per demo user

`User` and `Dependency` were the only seeded models still using Prisma's `@default(cuid())` instead of a fixed ID, and all three demo users shared a single `hashPassword()` call — meaning identical salts across all three rows, which defeats salting's purpose (one cracked hash exposes all three) and isn't representative of how real user rows are ever created.
Fix: added `DEMO_USER_IDS` (`USER-PM`, `USER-ENG-LEAD`, `USER-EXEC`) and `DEPENDENCY_IDS` (`DEP-001`..`DEP-008`) to `packages/core/src/seed/ids.ts`; `prisma/seed.ts` now calls `hashPassword()` once per user and passes explicit `id` values for both models. No schema change was needed — `@default(cuid())` still applies for any future non-seeded row, seed data just always supplies its own ID.

## 2026-07-18 (correction pass) — RecordType: one Prisma enum plus three Zod allowlists, not three Prisma enums

`RecordType` was reused as-is across three different contexts (evidence citations, proposed-change targets, audit targets) with no restriction preventing, say, a `ProposedChange` from targeting a `Supplier`. It also had no `DEPENDENCY` entry and no way to represent an audit event about a workflow record (an `ImpactAnalysis` starting, a `Decision` being recorded).
Decision: expanded `RecordType` to a full superset (added `DEPENDENCY`, `IMPACT_ANALYSIS`, `MITIGATION_OPTION`, `PROPOSED_CHANGE`, `DECISION`, `SOURCE_REFERENCE`), and added `packages/core/src/record-types.ts` with three Zod allowlists — `evidenceRecordTypeSchema` (domain records only), `proposedChangeTargetTypeSchema` (`MILESTONE`/`RISK`/`BUDGET_ITEM` only, matching `ProposedChangeType`), `auditTargetTypeSchema` (the full superset) — that Phase 4/5 application code must validate against when creating these rows.
Why one enum + Zod over three separate Prisma enums: Prisma has no native "enum subset" constraint either way, so per-context enforcement always happens at the application boundary regardless of which schema shape is chosen. Three near-identical Prisma enums would need to be kept in sync by hand (e.g. `PROGRAM_EVENT` belongs in both the evidence and audit sets but not the change-target set) — one source of truth plus explicit allowlists was judged less error-prone for an MVP, at the cost of the database itself not enforcing the restriction (only the application layer does, once Phase 4/5 wires these schemas into the actual create paths — they exist now but are not yet called from anywhere, since no code creates these rows yet).
Also added in the same migration: a `SourceReference` uniqueness constraint on `(impactAnalysisId, recordType, recordId)` (the same record must never be double-cited as evidence for one analysis) and a `Decision.traceId` index.
Migration: `20260718055852_expand_record_type_and_add_constraints`, applied to both `missionthread_dev` and `missionthread_test`.

## 2026-07-18 (correction pass) — ImpactAnalysis attempt lifecycle

Documenting the existing schema design rather than changing it: one `ImpactAnalysis` row represents one provider attempt for one program event. `ProgramEvent` has a one-to-many relation to `ImpactAnalysis` (re-analysis creates a new row), and each row carries its own `attempt` number, `traceId`, `status`, and duration/validation metadata. The "exactly one retry" rule (SPEC.md §10) is implemented by the Phase 4 orchestration code creating at most two `ImpactAnalysis` rows per triggering call (attempt 1, then attempt 2 only on failure) — not by a separate attempts table or by mutating one row in place, which would lose the failed attempt's own record. This satisfies SPEC.md §16's observability requirements (trace ID, attempt number, duration, safe error category are all columns on this same row) without introducing a second model before Phase 4 exists to populate it.

## 2026-07-18 (correction pass) — Dockerfile repaired: Next.js standalone output, non-secret build-time DATABASE_URL

The original Dockerfile's build stage ran `prisma generate` with no `DATABASE_URL` at all — `packages/core/prisma.config.ts` reads it eagerly, so this failed immediately. Its runtime stage also ran `npm run start --workspace @missionthread/web` without ever copying the root `package.json`, which npm workspace commands require.
Fix: enabled `output: "standalone"` and set `outputFileTracingRoot` to the monorepo root in `apps/web/next.config.ts` (verified via Next's own docs — monorepo tracing defaults to the app's own directory and would otherwise miss the `@missionthread/core` workspace package). The build stage now sets a clearly non-secret, unreachable placeholder `DATABASE_URL` build arg (`postgresql://build:build@localhost:5432/build_placeholder`) — sufficient because `prisma generate` never opens a connection — and the runtime stage runs `node apps/web/server.js` directly against the traced, pruned `.next/standalone` bundle, which needs no `npm`/workspace command and therefore no root `package.json` at runtime at all. Also discovered and defended against: Next's standalone output additionally copies any `.env` file it finds on disk at build time into the bundle; `.dockerignore` already excludes `.env` from the build context, and an explicit `rm -f` after the build step is a second layer of defense against that specific behavior. Runs as a non-root `nextjs` user.
Verified live: `docker build` succeeded; a container from the image started; `GET /login` returned 200 with the expected sign-in form; `GET /` correctly redirected unauthenticated; container was stopped and the test image removed afterward. Noted for real deployments (not needed for this local verification): Auth.js v5 logs an `UntrustedHost` warning when the request's `Host` header doesn't match its trusted-host configuration — a real deployment needs `AUTH_TRUST_HOST=true` or an explicit `AUTH_URL` set via the orchestrator's runtime environment, not baked into the image.
Alternatives: remove the Dockerfile and defer entirely to Phase 8 (rejected — SPEC.md §17 lists `Dockerfile` as a required Phase 1 file, and the standalone-output fix was achievable and verifiable now).

## 2026-07-18 (second correction pass) — Exact (host, port, database) target tuples replace independent host/database allowlists

The previous guard checked host and database name independently, so it never validated the port at all — `localhost:5432/missionthread_dev` (a developer's _other_ local Postgres, not this project's Docker Compose instance on 55432) would have passed. Replaced `APPROVED_LOCAL_HOSTS` + `APPROVED_DEV_DATABASE_NAMES` + `APPROVED_TEST_DATABASE_NAMES` with a single `ApprovedDatabaseTarget` tuple type (`{ host, port, database, requiresCi? }`) and three fixed tuple lists: `LOCAL_DEV_TARGETS` (`localhost`/`127.0.0.1` : `55432` / `missionthread_dev`), `LOCAL_TEST_TARGETS` (same hosts/port / `missionthread_test`), and `CI_TEST_TARGETS` (`localhost` : `5432` / `missionthread_test`, `requiresCi: true`). `findApprovedDatabaseTarget()` requires every field to match exactly; callers pass only the subset of tuples relevant to that operation (e.g. `reset-test-db.ts` only ever sees `LOCAL_TEST_TARGETS`, never the dev tuples).
CI detection: `requiresCi: true` tuples only match when `process.env.CI === "true"` (exact string match, same discipline as the opt-in flag) — GitHub Actions sets this automatically for every job, so nothing has to set it explicitly in `.github/workflows/ci.yml`.
IPv6: intentionally not supported. Verified directly in Node that `new URL("postgresql://x@[::1]:5432/db").hostname` returns the bracketed literal `"[::1]"`, not `"::1"` — the previous `APPROVED_LOCAL_HOSTS` list contained the unbracketed form, which could never actually have matched a real parsed URL. Rather than fix that mismatch, IPv6 was dropped entirely: nothing in this project's Docker Compose config, CI workflow, or local tooling connects over IPv6, so adding it now would be untested surface with no real use case. A bracketed `"[::1]"` tuple can be added later if a genuine need appears.
Compose-internal host removed: the previous `APPROVED_LOCAL_HOSTS` included `"postgres"` (the Docker Compose service's in-network hostname) on the theory that a future containerized app service might use it. `docker-compose.yml` currently defines only the `postgres` service itself — no web application service exists that would ever resolve that hostname — so this was a dead allowlist entry. Removed; will be reintroduced if a real containerized runtime path is added (e.g. in Phase 8).
Migration: none required — this is an application-layer (Zod/TypeScript) change, not a schema change.

## 2026-07-18 (second correction pass) — Destructive commands renamed to make intent explicit at invocation, and the opt-in flag is never permanently enabled

The example env files previously shipped `ALLOW_DESTRUCTIVE_DATABASE_OPERATION=true` as enabled configuration, described as "safe to leave set" — which defeats the point of requiring an explicit opt-in, since it would never actually be off. Removed the flag from `.env.example` and `.env.test.example` entirely.
In its place: `packages/core`'s scripts were renamed to `db:seed:internal` and `db:reset:test:internal` (not meant to be run directly), and the root `package.json` gained `db:seed:destructive` and `db:reset:test`, both of which set the opt-in flag only for their own child process via a new `scripts/with-destructive-auth.mjs` wrapper. The wrapper uses `child_process.spawn()` with an explicit `env` object rather than shell `VAR=value cmd` syntax, so the same command works on Windows (where `cmd.exe` doesn't support that syntax) without adding a `cross-env`-style dependency. CI's "Seed CI test database" step now calls `npm run db:seed:destructive` — the flag is no longer set at the workflow-wide `env:` level.
Naming: avoided a root script and a workspace script sharing the exact same name (the previous `db:reset:test` at both levels was ambiguous about which one actually ran) — the workspace-level scripts now carry an explicit `:internal` suffix.
Alternatives: keep the flag permanently enabled in example files (rejected — the explicit finding this pass); add a `cross-env` dependency for shell-syntax portability (rejected — a ~20-line wrapper needed no new dependency).

## 2026-07-18 (second correction pass) — Deferred: `ProposedChangeType.NEW_ACTION` target-field conflict

Not implemented this pass (Phase 5 work) — documented so the conflict isn't rediscovered from scratch. `ProposedChange.targetRecordId` and `targetRecordType` are currently non-nullable, but `ProposedChangeType.NEW_ACTION` describes creating a new action, not modifying an existing record — there is no existing target to reference, so a `NEW_ACTION` row would be forced to either fabricate a target or violate the non-null constraint.
Planned Phase 5 solution: make `targetRecordId` and `targetRecordType` nullable in the schema; validate `ProposedChange` creation with a Zod discriminated union keyed by `changeType` — `MILESTONE_DATE`/`RISK_UPDATE`/`BUDGET_UPDATE` require both target fields (validated against `proposedChangeTargetTypeSchema`), `NEW_ACTION` forbids them, unless a future revision explicitly models actions as belonging to an existing parent record. Never fabricate a placeholder target ID to satisfy a non-null constraint.
No schema or migration change made now — `SPEC.md` does not require this before Phase 2, and implementing it without the Phase 5 workflow code that would actually create these rows risks guessing the wrong shape.

## 2026-07-18 (second correction pass) — Deferred: `ImpactAnalysis` retry-pair grouping key

Not implemented this pass (Phase 4 work). The existing one-row-per-attempt design (see the 2026-07-18 correction-pass entry above) correctly persists each attempt as its own row, but has no stable key linking attempt 1 and attempt 2 of the _same_ retry sequence together, distinct from a later, separate re-analysis of the same event.
Planned Phase 4 solution: add an `analysisRunId` column with a uniqueness rule equivalent to `(analysisRunId, attempt)`. Attempt 1 and its retry (attempt 2) share one `analysisRunId`; a later, independent re-analysis of the same `ProgramEvent` gets a new `analysisRunId`. This keeps failed attempts individually persisted (SPEC.md §10's "persist a failed analysis attempt" requirement) while making "the two attempts that belong together" a queryable fact instead of an inferred one.
Not creating a separate `AnalysisAttempt` model unless Phase 4 implementation proves `analysisRunId` insufficient — SPEC.md's own guidance is to avoid a new model unless it adds real value beyond what a grouping key provides.
No schema or migration change made now, for the same reason as the `NEW_ACTION` entry above.

## 2026-07-18 (third correction pass) — Exact GitHub Actions target requirement: `GITHUB_ACTIONS=true`, not `CI=true`

The second-pass CI target tuple authorized on `CI === "true"`. That's insufficient specifically for a _GitHub-Actions_-only target: `CI=true` is a loose convention many local shells, editor integrations, and other CI providers also set, so it can't uniquely identify "this is a GitHub Actions runner" the way `GITHUB_ACTIONS` (which only GitHub Actions sets) can.
Fix: renamed `ApprovedDatabaseTarget.requiresCi` → `requiresGitHubActions`, and `CI_TEST_TARGETS` → `GITHUB_ACTIONS_TEST_TARGETS`; `findApprovedDatabaseTarget()` now checks `env.GITHUB_ACTIONS === "true"` exclusively for that tuple — `CI` is not consulted at all, so `CI=true` alone (GitHub Actions absent) correctly fails, and `GITHUB_ACTIONS=true` alone (CI absent) correctly passes. 16 new/updated unit tests cover both variables independently, uppercase/whitespace variants of the value, and the dev-database-on-port-5432 and arbitrary-host/port negative cases.
Alternatives: require both `GITHUB_ACTIONS=true` and `CI=true` together (rejected — `GITHUB_ACTIONS` alone is already authoritative and sufficient; requiring both would only make the check more likely to fail in a legitimate GitHub Actions run if some future workflow change ever left `CI` unset for another reason).

## 2026-07-18 (third correction pass) — Target-specific destructive seed commands; explicit `MISSIONTHREAD_SEED_SCOPE`

`seed.ts`'s `clearExistingData()` previously accepted the union of `LOCAL_DEV_TARGETS`, `LOCAL_TEST_TARGETS`, and the GitHub Actions target all at once, and the single public command `db:seed:destructive` authorized whichever one `DATABASE_URL` happened to point at — every possible target was allowlisted, but the command name itself never said which database a user was actually authorizing.

## 2026-07-18 (fourth correction pass) — Seed scope validated before database configuration is loaded or Prisma is initialized

Previously, `seed.ts` validated `MISSIONTHREAD_SEED_SCOPE` inside `clearExistingData()`, which ran after the module had already loaded `.env`/`.env.test` via `dotenv` and already constructed a live `PrismaPg` adapter and `PrismaClient` against `DATABASE_URL`. An invalid or missing scope was still caught before any row was deleted, but only after database configuration had already been loaded and the Prisma adapter/client had already been constructed — an unnecessary sequencing gap for a check this cheap to run first. (Constructing `PrismaPg`/`PrismaClient` does not itself open a network connection — Prisma connects lazily on first query — so the prior ordering did not necessarily reach the database; the gap was in configuration/object construction happening before validation, not in an actual connection being opened.)

Consequence: an invalid `MISSIONTHREAD_SEED_SCOPE` now fails before this process loads database configuration or constructs the Prisma client, not merely before it deletes data — and the failure message never echoes back whatever was actually in the environment variable.
Alternatives: leave the check where it was, inside `clearExistingData()` (rejected — the fail-first ordering guarantee this pass is enforcing is that an unauthorized process never even reaches the point of configuring database access, not just the point of mutating data); keep printing the raw rejected value for debuggability (rejected — an environment variable that fails validation is, by definition, unexpected input, and treating it as safe to log is exactly the assumption that leads to credential leaks in error paths).

## 2026-07-18 (Phase 2) — Deterministic-service error strategy: one `ServiceResult<T>` union, never throw for expected failures

Why: `SPEC.md` §8 lists eleven public functions with heterogeneous failure modes (missing component, missing risk, invalid input array). Mixing thrown exceptions for some and return values for others would force every caller (Phase 3 UI, Phase 4 evidence builder) to remember which functions throw and which don't.
Decision: every function in `packages/core/src/analysis/` returns `ServiceResult<T> = { ok: true; data: T } | { ok: false; error: DomainError }` (`packages/core/src/analysis/types.ts`). `DomainError` has `code: "NOT_FOUND" | "VALIDATION_ERROR"`, an optional `entityType`/`entityId`, and a safe `message`. Raw Prisma errors are never returned to callers — `NOT_FOUND` is produced by an explicit existence check, not by catching a Prisma "record not found" exception. Input validation (`packages/core/src/analysis/schemas.ts`) runs before any database query and produces `VALIDATION_ERROR` the same way.
Alternatives: throw typed errors (rejected — forces try/catch at every call site for what are often expected, non-exceptional outcomes, e.g. "this component has no impacted requirements" vs. "this component doesn't exist" are both ordinary results, not exceptions); throw for NOT_FOUND but return a value for validation (rejected — the whole point is one consistent strategy).

## 2026-07-18 (Phase 2) — Input-ID validation: reject padding/whitespace/duplicates, never silently normalize

`entityIdSchema` (`packages/core/src/analysis/schemas.ts`) rejects an empty string, a whitespace-only string, and a string with leading/trailing whitespace — it does not trim-and-accept. Real seeded IDs (`COMP-EC440`, `REQ-001`, ...) never have surrounding whitespace; silently trimming a caller's malformed ID would hide a caller-side bug instead of surfacing it as a `VALIDATION_ERROR`.
For functions taking an ID array (`getVerificationGaps`, `getRelatedDefects`), the array schema additionally rejects duplicate entries outright (`VALIDATION_ERROR`), rather than silently deduplicating — a duplicate requirement ID would double-count that requirement in coverage ratios and gap lists, which is exactly the "distort results" case `PROJECT_GUIDE.md`'s Phase 2 instructions call out. An empty array is accepted (returns an empty result), since "analyze zero requirements" is a valid, non-malformed call.

## 2026-07-18 (Phase 2) — Dependency traversal direction: `fromMilestone` is upstream, `toMilestone` is downstream; both returned separately

The `Dependency` model's own schema comment says "from finishes before to starts." `getDependencyChain(milestoneId)` therefore treats `fromMilestoneId` edges as pointing to a **prerequisite** (upstream) and `toMilestoneId` edges as pointing to a **dependent** (downstream), and returns both as separate arrays (`upstream`, `downstream`) rather than picking one direction — a caller asking "what does this milestone depend on" and a caller asking "what depends on this milestone" both need an answer, and conflating them would lose information.
Cycle/duplicate protection is implemented in a pure, DB-free helper (`traverseDependencyChain()` in `packages/core/src/analysis/dependencies.ts`) that takes an in-memory edge list and a start ID: a `visited` set keyed by milestone ID stops traversal from ever revisiting a node (so a synthetic cycle terminates instead of looping), and a `visitedEdges` set keyed by `${from}->${to}` prevents a duplicate edge from producing a duplicate chain entry. The DB-backed `getDependencyChain()` fetches every `Dependency` row for the milestone's program in one query (not one query per hop — see the N+1 note below) and hands the resulting edge list to the pure helper. This also makes cycle protection unit-testable with a fabricated cyclic edge list, without needing to seed an actual cycle into Postgres.

## 2026-07-18 (Phase 2) — Schedule cascade assumption: a delay shifts every impacted milestone's planned date by the same day count

`calculateScheduleExposure(eventId)`'s `latestExposedDate` is computed as `max(milestone.plannedDate + directDelayDays)` across every impacted milestone (direct: milestones on the event's component; dependency-derived: downstream milestones reached via `getDependencyChain`'s downstream edges from each direct milestone). This is a deliberate simplification, not a claim that real schedule float/slack was modeled: the schema has no per-milestone float/buffer field, so a 1:1 "the whole delay propagates downstream unmitigated" assumption is the only value derivable without inventing data. This is recorded as an explicit assumption in `buildAnalysisEvidence()`'s `assumptions` list, not presented as a precise scheduling-engine result.
Date arithmetic uses UTC calendar days: seeded/stored dates are date-only values (`new Date("2026-09-15")`, parsed by JS as UTC midnight), so `Math.round((b.getTime() - a.getTime()) / 86_400_000)` gives an exact calendar-day count with no local-timezone drift, as long as both operands are UTC-midnight-aligned — true for every date this schema stores. `calculateScheduleExposure` also compares the event's stored `delayDays` against the value computed from `originalDate`/`revisedDate` and reports (does not silently swallow) any disagreement.

## 2026-07-18 (Phase 2) — Budget variance sign convention and mixed-currency handling

Sign convention: `varianceAmount = actualTotal - plannedTotal` throughout `calculateBudgetVariance`/`calculateBudgetExposure`. A positive value is a cost overrun (spent more than planned); negative is an underrun. All arithmetic uses `Prisma.Decimal` (re-exported from `@prisma/client`, backed by `decimal.js`), never plain JS `number` addition, so cents can't drift from binary floating-point rounding; totals are serialized as fixed-2-decimal strings (`"480000.00"`), not numbers, in the returned result.
`BudgetItem.currency` defaults to `"USD"` but isn't constrained to a single value program-wide. `calculateBudgetVariance` groups items by currency: if exactly one currency is present, the top-level `plannedTotal`/`actualTotal`/`varianceAmount`/`variancePercentage` are populated for it; if the program's budget items span more than one currency, those top-level fields are `null` (never a silently-wrong cross-currency sum) and a `byCurrency` breakdown array carries the same fields per currency instead. `variancePercentage` is `null` (not `Infinity`/`NaN`) whenever `plannedTotal` is exactly zero, with a note in `missingData`.

## 2026-07-18 (Phase 2) — Budget exposure cannot compute a new incremental delay cost; documents the schema gap instead of inventing one

`calculateBudgetExposure(eventId)` links the event to budget items only through `ProgramEvent.componentId -> BudgetItem.componentId` — the only deterministic link the schema provides. Neither `ProgramEvent` nor `BudgetItem` has a per-day or per-unit cost-impact field, so there is no way to deterministically compute "the dollar cost of this specific 28-day delay" — doing so would require inventing a number. Instead, `totalDeterministicExposure` is defined as the sum of `plannedAmount` across the directly-linked budget items (the total budget tied to the affected component, i.e. the amount actually at risk from schedule/cost pressure on that component), and `currentVarianceTotal` is the existing `actual - planned` variance already present on those same items (same sign convention as `calculateBudgetVariance`). `missingData` always includes an explicit note that no incremental delay-cost figure is calculable from current schema fields. This limitation is also carried into `buildAnalysisEvidence()`'s `unknowns` list.

## 2026-07-18 (Phase 2) — Risk score formula, band mapping, and a genuine seeded inconsistency (`RISK-003`)

Formula: `score = probability * impact` (both stored as `Int`, conventionally 1–5 in this dataset, giving a score range of 1–25). Band mapping: `1–4` → `LOW`, `5–9` → `MEDIUM`, `10–14` → `HIGH`, `15–25` → `CRITICAL`. `calculateRiskScore` reports the computed band alongside the stored `Risk.severity` and a `severityConsistent: boolean` — it never lets the stored text label silently override the numeric result.
Applying this mapping to the seed data surfaces a real, pre-existing inconsistency: `RISK-003` (`probability: 2, impact: 2` → score `4` → band `LOW`) is stored with `severity: MEDIUM`. This is not a bug introduced by Phase 2 — it's the kind of disagreement the spec explicitly asked this function to detect and report rather than paper over, and `db-safety.test.ts`-style regression coverage exists for exactly this case (`RISK-003` is asserted to come back `severityConsistent: false`). Probability/impact values outside the expected 1–5 range are not rejected (they're read from the database as-is) but are surfaced via a `warnings` array rather than silently accepted as normal.

## 2026-07-18 (Phase 2) — Readiness score: five equal-weighted, independently-documented factors, 0–100, missing data scored as neutral (100%) with a warning

`calculateReadinessScore(programId)` is `round(sum of five factors)`, each worth 20 points (100 total), each computed as `(healthyCount / totalCount) * 20`:

1. **Verification coverage** (20 pts): proportion of the program's requirements with zero verification gaps (every linked test `PASSED`, and at least one test exists) per `getVerificationGaps`.
2. **Test health** (20 pts): proportion of the program's test cases with `outcome: PASSED`.
3. **Milestone health** (20 pts): proportion of milestones whose `status` is not `AT_RISK` or `DELAYED`.
4. **Defect health** (20 pts): proportion of defects whose `status` is not `OPEN` or `IN_PROGRESS` (i.e. `RESOLVED`/`CLOSED` count as healthy).
5. **Risk health** (20 pts): `(1 - averageActiveRiskScore / 25) * 20`, where "active" risks are `status: OPEN` or `MITIGATING` (a `CLOSED` risk no longer counts against readiness) and `25` is the theoretical max `calculateRiskScore` (5×5), not just the observed max in the data — so the scale doesn't silently shift if worse risks are added later.

Missing-data policy: if a factor's denominator is zero (e.g. a program with no budget items, no tests, no risks), that factor scores its **full** 20 points (treated as neutral/healthy, not penalized for absent data) and a corresponding entry is added to the result's `warnings` array — this was chosen over redistributing the other factors' weights (extra complexity for a 5-factor MVP formula) or scoring it zero (would wrongly penalize a program for not yet having, say, any defects logged). Rounding: the total is rounded to the nearest integer (`Math.round`) and clamped to `[0, 100]`; factor subscores are reported unrounded in the breakdown so the total is reproducibly re-derivable from them.
Budget variance was deliberately **not** made a sixth factor: `calculateBudgetVariance` can return `null` totals under mixed currency, and folding a nullable input into a fixed 100-point formula would need its own missing-data special case on top of the general one above — deferred rather than guessed at for the MVP.

## 2026-07-18 (Phase 2) — Evidence bounding: `rawNotes` is never a `summary`; it's a separate, explicitly-labeled untrusted field

`buildAnalysisEvidence(eventId)`'s `evidence[]` array (the bounded allowlist an eventual Phase 4 AI call would receive) only ever contains `summary` text built from validated, structured fields (titles, statuses, dates, IDs) — **never** `ProgramEvent.rawNotes`, which is supplier-submitted, untrusted free text (the seeded `EVT-SUPPLIER-001` row deliberately contains a prompt-injection-style sentence — see `prisma/seed.ts`). Putting `rawNotes` into an evidence item's `summary` field would make it structurally indistinguishable from every other trusted, deterministically-derived summary once it reached a future prompt.
Instead, `buildAnalysisEvidence()`'s top-level result carries a separate `untrustedSupplierNotes: string | null` field, outside the `evidence[]` array and outside `EVIDENCE_RECORD_TYPES`, explicitly documented as data — never instructions — that Phase 4 must keep clearly isolated (e.g. wrapped and labeled in its own prompt section) and that no Phase 2 calculation ever reads. Evidence items are deduplicated by `recordType + recordId` and sorted by a fixed `EVIDENCE_RECORD_TYPES` order then by `recordId`, so evidence output is stable and reproducible across calls.

**Superseded by the 2026-07-18 (Phase 2 correction pass) entries below** — `untrustedSupplierNotes` was replaced by a two-field `untrustedText: { reason, rawNotes }`, since `event.reason` turned out to have the same untrusted-free-text problem and was still leaking into the trusted event summary. Kept here as the historical record of the original design.

## 2026-07-18 (Phase 2 correction pass) — `buildAnalysisEvidence` now returns the full structured deterministic analysis, not just source-reference summaries

## 2026-07-18 (Phase 2 correction pass) — Computed schedule data is authoritative in `eventFacts`; `event.reason` joins `rawNotes` as untrusted text

Confirmed as a real, reproducible defect (two related issues): (1) the previous evidence summary interpolated `event.delayDays` directly, presenting the _stored_ value as fact even on the (rare, but explicitly detected elsewhere in Phase 2) occasion it disagrees with the dates; (2) `event.reason` — free text a user enters when recording an event, no different in kind from `rawNotes` — was embedded directly into the trusted `PROGRAM_EVENT` evidence summary, while only `rawNotes` was treated as untrusted.

## 2026-07-18 (Phase 2 correction pass) — Explicit, documented evidence bounds

Not a defect in existing behavior (the seeded data never came close to any reasonable limit), but a real gap: nothing in `buildAnalysisEvidence()` bounded the size of what it returned, which the task correctly identified as unsafe to leave unbounded before this evidence could ever be handed to a model (Phase 4).
Limits (`EVIDENCE_LIMITS` in `packages/core/src/analysis/evidence.ts`): 100 maximum total evidence items, 25 maximum items per record type, 500-character maximum evidence-item summary length, 4,000-character maximum length per untrusted-text field (`reason`, `rawNotes` independently). Applied in `applyEvidenceBounds()` (item limits, pure and exported) and `truncateText()`/`truncateUntrustedField` (length limits, pure and exported): summaries are truncated first, then the per-record-type cap, then the overall cap — each preserving the existing deterministic order (`EVIDENCE_RECORD_TYPES` order, then `recordId`) rather than re-sorting, and each producing an `unknowns` entry rather than a silent drop. `truncateText()` is surrogate-pair-safe: it never cuts a UTF-16 astral character (e.g. an emoji) in half, checking whether the code unit immediately before the cut point is an unpaired high surrogate and backing up one further position if so.
Alternatives: add a token-counting/tokenizer dependency now to bound by estimated model tokens instead of character/item counts (rejected — the task explicitly defers this past Phase 2; character and item counts are simple, dependency-free, and already meaningfully bound the payload); silently drop overflow without a note (rejected — indistinguishable from "there was nothing more to report," which is a materially different and more dangerous claim for an evidence set headed toward a model).

## 2026-07-18 (Phase 3) — Fixed a real CI blocker: the smoke test could hang for GitHub's full 6-hour job timeout

Confirmed as a real, reproducible defect, discovered while verifying CI was green before starting Phase 3: the smoke-test step of the immediately preceding push (`e7096d1`) never completed — GitHub Actions job history showed several earlier pushes auto-cancelled after exactly 6 hours (GitHub's default job timeout), each apparently stuck at the same step, with no failure signal at all in between.

## 2026-07-18 (Phase 3) — Event-entry contract: a strict Zod discriminated union, server-computed `delayDays`

`packages/core/src/events/schemas.ts` defines `eventEntrySchema`, keyed by `eventType` (`SUPPLIER_DELAY` | `GENERAL_UPDATE`), each branch `.strict()` (rejects unknown fields, including a client-supplied `delayDays` — which was never part of the contract in the first place, not merely rejected after the fact). Dates use a custom `dateOnlySchema`: a `YYYY-MM-DD` format check plus a round-trip `Date.UTC(...)` calendar-validity check, so `"2026-02-30"` (matches the format regex but isn't a real date) is rejected. `delayDays` itself is computed server-side in `recordProgramEvent()` via the same `utcDayDifference()` already used by `calculateScheduleExposure()` (Phase 2) — one shared UTC-day-arithmetic implementation, not two.
Bounds: `MAX_QUANTITY = 100_000` (arbitrary but documented — large enough for any real lot size, small enough to catch an obviously-wrong input), `MAX_REASON_LENGTH = 500`, `MAX_RAW_NOTES_LENGTH = 4_000` (independent of, though numerically equal to, `EVIDENCE_LIMITS.maxUntrustedTextLength` — one governs what a caller may submit, the other governs later truncation for a future model input; they're allowed to drift independently).
Alternatives: validate dates with a date-parsing library (rejected — a regex plus one `Date.UTC` round-trip check is sufficient and adds no dependency); allow `GENERAL_UPDATE` to accept supplier-delay fields "just in case" (rejected — the task explicitly calls for rejecting them unless deliberately supported and documented, which they are not).

## 2026-07-18 (Phase 3) — Mutation authorization: `FORBIDDEN` extends the Phase 2 error strategy; database role is always re-verified, never trusted from a session claim

Extended `DomainErrorCode` (`packages/core/src/analysis/types.ts`) from `"NOT_FOUND" | "VALIDATION_ERROR"` to add `"FORBIDDEN"`, with a matching `forbidden()` constructor — the same `ServiceResult<T>` shape every Phase 2 service already returns, not a second error strategy for mutations.

## 2026-07-20 (Phase 4) — Retry policy is uniform across AI modes, not gated on `AI_MODE`

`orchestrator.ts`'s attempt loop (max 2 attempts, retry exactly once on a retryable failure category) has no `if (aiMode === "live")` branch — it retries identically regardless of which provider produced the failure. Persists `status: SUCCEEDED` directly (never runs through the `PENDING` transient state, which only has meaning during a live `runImpactAnalysis()` call) since a fresh seed should demonstrate a completed analysis immediately.
A real ESM ordering bug was found and fixed while building this: `buildAnalysisEvidence()`/`packages/core/src/ai` transitively import `packages/core/src/db.ts`, whose module-level `PrismaClient` is constructed from `process.env.DATABASE_URL` at import-evaluation time. `seed.ts` already has its own top-level `dotenv` load (guarded by `if (!process.env.DATABASE_URL)`), but ES module `import` statements are evaluated in source order _before_ the importing module's own top-level statements run, regardless of where in the file the `import` is textually written — so a static import of these modules would have constructed `../db.ts`'s client before `DATABASE_URL` was set, silently connecting to whatever default-local Postgres the driver falls back to (observed directly: a `P2021 Table does not exist` error against a database that, on inspection, was neither `missionthread_dev` nor `missionthread_test`). Fixed by making these two imports dynamic (`await import(...)`) inside `seedImpactAnalysis()` itself, which only evaluates at the point it's actually awaited — by which time `main()`, and therefore the `dotenv` load, has already run. `seed.ts`'s pre-existing imports of `../src/seed/ids` and `../src/db-safety` were never affected by this, since neither transitively imports `../db.ts`.
New fixed seed IDs (`packages/core/src/seed/ids.ts`): `ANALYSIS_IDS`, `ANALYSIS_RUN_IDS`, `ANALYSIS_TRACE_IDS`, `MITIGATION_OPTION_IDS` (always exactly 3), `ANALYSIS_AUDIT_EVENT_IDS`. `SourceReference` row IDs are deliberately _not_ a pre-enumerated fixed list — they're derived deterministically at seed time from each cited record's own `(recordType, recordId)`, since the exact evidence-citation set is itself already fully deterministic seed data; pre-enumerating a parallel ID list whose length must exactly match the pipeline's real output would be fragile for no real benefit.
Alternatives: hand-author the seeded analysis's executive summary/mitigation options directly as static seed data (rejected — the task explicitly requires reusing the same validated mock-output fixture the pipeline itself produces, not a separate duplicate shape); run the seeded analysis through the full `runImpactAnalysis()` orchestrator instead of calling the lower-level functions directly (rejected — the orchestrator's authorization/attempt-lifecycle machinery has no meaning at seed time, when there's no authenticated actor session; calling `buildAnalysisEvidence`/`buildModelInputProjection`/`generateMockImpactAnalysis` directly is the minimal real subset that still proves the pipeline produces valid, schema-passing output).

## 2026-07-20 (Phase 4) — Non-interactive Prisma migration generation via a throwaway shadow database

`prisma migrate dev` requires an interactive terminal and refused to run in this environment ("Prisma Migrate has detected that the environment is non-interactive"). The non-interactive alternative, `prisma migrate diff --from-migrations ... --to-schema ... --script`, itself requires a shadow database to compute the diff, which needs `datasource.shadowDatabaseUrl` configured in `prisma.config.ts` — that key was added _temporarily_, generation was run once against a throwaway `missionthread_shadow` database (created via `psql`, on the same local Postgres instance, immediately before) with `SHADOW_DATABASE_URL` set only for that one command, and the config key was then removed again immediately afterward. `shadowDatabaseUrl` was not left configured permanently: `@prisma/config`'s `env()` helper throws if the named variable is unset, and `SHADOW_DATABASE_URL` is not part of this project's normal environment (`.env`/`.env.test`/CI) — leaving the key in place would have broken every other Prisma command (`validate`, `generate`, `migrate deploy`, the app's own runtime) the moment that variable was absent. The shadow database itself was dropped immediately after the migration SQL was generated.
The generated migration was then applied with `prisma migrate deploy` (which needs no shadow database and works non-interactively), against both `missionthread_dev` (to verify the seed pipeline) and, later, `missionthread_test` (via the existing `db:reset:test` command, under fresh per-invocation destructive-operation authorization, since the test database's schema needed the same migration before the new AI test suite could run against it).
Alternatives: hand-write the migration SQL instead of generating it (rejected — this specific change is simple enough that hand-writing was tempting, but generating it directly from the schema diff removes an entire class of hand-transcription error, and the generated SQL was reviewed before being applied); leave `shadowDatabaseUrl` configured permanently pointing at a real, always-present shadow database (rejected — no such database exists in this project's Docker Compose setup, and adding one is a bigger, unrequested change to the local dev environment for a capability only needed at migration-authoring time).

## 2026-07-20 (Phase 4) — Live AI mode is unverified against the real OpenAI API in this repository

By hard constraint (`PROJECT_GUIDE.md`, and repeated explicitly in this phase's authorization): no test, integration test, smoke test, or CI step is permitted to call the live provider, and a real, paid live-mode request may only ever be made by a developer who separately supplies their own `OPENAI_API_KEY` and explicitly authorizes the cost — never by this agent, and never automatically. Consequently, `openai-provider.ts`'s Responses API request construction, its strict-JSON-schema `text.format` configuration, and its error classification were built and unit-tested only against the `openai` npm package's own published TypeScript types and documentation (via an injected fake `client` — see `openai-provider.ts`'s constructor), never against a real API response. This is a genuine, currently-open limitation, not an oversight: it is the direct and correct consequence of the constraint above, and is recorded here (and in `README.md`'s Limitations section) rather than silently left unstated.

## 2026-07-21 (Phase 4 correction pass) — `mitigationOptions` output field: `z.array(...).length(3)` instead of `z.tuple([...])`

Confirmed as a real, reproducible defect (not yet caught because the live provider had never actually been exercised): `impactAnalysisOutputSchema.mitigationOptions` was a `z.tuple([opt, opt, opt])`. `z.toJSONSchema()` converts a Zod tuple to JSON Schema's `prefixItems` (positional-item array validation) — a keyword outside OpenAI Structured Outputs' documented supported subset. A live request built with the original schema would very likely have been rejected by the API (or silently fallen back to non-strict enforcement), and this was never caught locally because no automated test may call the live provider (see the entry directly above) and the mock provider never exercises the JSON-schema-generation path at all.
Fix: `mitigationOptions: z.array(mitigationOptionOutputSchema).length(3, "exactly three mitigation options are required")`. `z.array(...).length(n)` converts to `{ type: "array", items: <schema>, minItems: n, maxItems: n }` — verified directly (`z.toJSONSchema()` output inspected before and after) — which expresses the identical "exactly three" constraint using only `items`/`minItems`/`maxItems`, all of which are in OpenAI's supported subset. Zod's own runtime validation is unweakened: `.length(3)` still rejects any array whose length isn't exactly 3 (verified: 2 elements and 4 elements both fail `safeParse`, exactly 3 passes), and the pre-existing `.refine()` requiring exactly one `isRecommended: true` is untouched.
Type consequence: `ImpactAnalysisOutput["mitigationOptions"]` is now `MitigationOptionOutput[]` rather than a compile-time 3-tuple. No caller relied on tuple-specific typing (all iterate with `.map()`/`.entries()`/`for...of`), so this had no other code impact.
Alternatives: keep the tuple and manually strip `prefixItems` from the generated JSON Schema before sending it (rejected — would mean the JSON Schema sent to the provider no longer accurately reflects an array of exactly 3 homogeneous items, replacing it with something that isn't equivalent JSON Schema for that constraint); switch to `z.tuple([...]).length ...` (not a real Zod API — tuples don't support `.length()`).

## 2026-07-21 (Phase 4 correction pass) — Provider-facing JSON Schema subset: generated from the authoritative schema, then verified, not hand-maintained

`packages/core/src/ai/openai-schema.ts` is new: `buildOpenAiImpactAnalysisJsonSchema()` generates a JSON Schema from `impactAnalysisOutputSchema` via `z.toJSONSchema()` (unchanged from the original Phase 4 approach — still the single authoritative source, never a second hand-maintained schema) and then recursively verifies the result with `assertOpenAiCompatibleJsonSchema()` before returning it. The verifier checks two things throughout the whole schema tree: (a) none of `prefixItems`, `unevaluatedItems`, `contains`, `minContains`, `maxContains`, `propertyNames`, `patternProperties` appear anywhere — keywords draft-2020-12 permits but OpenAI Structured Outputs strict mode does not document support for; (b) every object-typed schema node declares `additionalProperties: false` with every one of its `properties` listed in `required` (OpenAI strict mode has no concept of an optional property — this codebase already expresses optionality via `.nullable()`, never `.optional()`, specifically for this reason, unchanged from the original Phase 4 design).
Chose **verify, not silently normalize**: if `z.toJSONSchema()` ever produced a disallowed keyword in the future (e.g., a future field added via `z.record()` or `z.string().regex()` combined with a schema shape that emits `patternProperties`), this throws a clear, located error at module load time rather than silently stripping or rewriting the offending keyword — a schema this function had to "fix" would mean the JSON Schema actually sent to the provider no longer matches what `impactAnalysisOutputSchema` really requires, which is a correctness bug worth failing loudly on. `openai-provider.ts` now imports `buildOpenAiImpactAnalysisJsonSchema()` instead of calling `z.toJSONSchema()` inline.
Known residual risk, documented rather than guessed away: OpenAI's own documented "supported subset" isn't published as a machine-checkable spec, so this allowlist-of-disallowed-keywords was assembled from Structured Outputs documentation and cannot be verified against a real API response in this repository (see "Live AI mode is unverified..." above). If a real live run is ever performed and OpenAI rejects the schema for a reason not on this list, the fix is to extend `OPENAI_DISALLOWED_JSON_SCHEMA_KEYWORDS`, not to bypass the check.
Alternatives: validate the schema only informally via manual inspection before each release (rejected — the task requires this be directly testable, and a regression here would only be caught by an actual failed live call, which this project is expressly forbidden from making in automated contexts); maintain a fully separate, hand-written OpenAI-specific schema (rejected — reintroduces exactly the drift risk the original Phase 4 design (generate from the authoritative schema) was built to prevent).

## 2026-07-21 (Phase 4 correction pass) — Complete attempt-evidence persistence: every supplied record, not just the cited subset

Confirmed as a real design gap: the original Phase 4 persistence only ever created a `SourceReference` row for IDs appearing in the model's own `sourceRecordIds`/option `sourceRecordIds` — i.e., only the subset the model chose to cite. The full evidence allowlist an attempt was actually built from (`ModelInputProjection.evidenceAllowlist`) was never itself retained anywhere once the attempt completed; a failed attempt retained nothing at all, since the old `persistFailedAttempt()` never touched `SourceReference`.
Fix: `SourceReference` gained two columns — `wasCited Boolean @default(false)` and `citationContexts Json?` (bounded, fixed-vocabulary strings only: `"analysis"` for a top-level citation, `"option:<index>"` for a per-mitigation-option citation — never free-form model text). Migration `20260720000000_add_evidence_snapshot_and_readiness_snapshot`.
New pure helper, `buildAttemptSourceReferenceSnapshot(modelInput, output?)` (`packages/core/src/ai/attempt-persistence.ts`): with no `output`, returns every allowlisted item exactly once, all `wasCited: false` — the complete "supplied" snapshot. With `output` supplied, additionally marks which items were cited and in which context(s) (an item can appear in multiple contexts, e.g. both the top-level analysis and one option). Defensively throws if `output` cites an ID that doesn't resolve to any allowlist entry — should be unreachable in practice, since `output-validation.ts`'s semantic validation already rejects such an output before it ever reaches this function, but the invariant is checked, not assumed. Never includes `untrustedData` content in any summary or citation context — proven by a dedicated test asserting the seeded event's prompt-injection-style sentence never appears in a serialized snapshot.
Orchestrator (`orchestrator.ts`) restructured around this: a new `persistPendingAttempt()` runs one transaction, per attempt, before the provider is ever called — creates the `PENDING` `ImpactAnalysis` row, **every** `SourceReference` row from the no-`output` snapshot, and the `ANALYSIS_STARTED` audit event. `persistSucceededAttempt()`'s transaction no longer _creates_ `SourceReference` rows (they already exist) — it _updates_ the cited subset's `wasCited`/`citationContexts` via the `(impactAnalysisId, recordType, recordId)` compound unique key. A failed attempt's rows are never revisited after `persistPendingAttempt()`, so they correctly retain the complete supplied snapshot, all `wasCited: false` — verified directly (`orchestrator.test.ts`, "[failed attempt retains its complete supplied evidence snapshot]"). A retry (attempt 2) gets its own `persistPendingAttempt()` call against the same `modelInput`, producing an entirely separate set of `SourceReference` rows scoped to its own `impactAnalysisId` — verified directly that two attempts in one run never share a `SourceReference` row ID.
`prisma/seed.ts`'s `seedImpactAnalysis()` now calls the same `buildAttemptSourceReferenceSnapshot()` helper (imported from `../src/ai`, dynamically for the same ESM-ordering reason documented in the original Phase 4 entry) instead of its own ad hoc cited-only loop — verified by a dedicated test that independently rebuilds the evidence/model-input pipeline for the seeded event and confirms the persisted rows match what the shared helper produces.
Consequence for the seeded demonstration analysis specifically: the mock provider's citation strategy (cite up to 30 evidence items) happens to cite all 24 of `EVT-SUPPLIER-001`'s evidence records, so the seeded fixture has zero uncited rows today — this is a property of that particular event's evidence volume, not a limitation of the underlying mechanism, which is proven correct against a deliberately partial-citation scripted provider in `orchestrator.test.ts`.
Alternatives: keep a cited-only `SourceReference` table and add a separate `EvidenceSnapshot` model for the full supplied set (rejected — the task explicitly directs against a new model unless `SourceReference` is proven unable to represent the requirement; two boolean/JSON columns were sufficient); persist the full snapshot only in a log line rather than the database (rejected — a structured log isn't queryable from the analysis workspace UI, and the whole point is to make the distinction visible there).

## 2026-07-21 (Phase 4 correction pass) — Cited vs. supplied-only evidence, surfaced distinctly in the UI

The analysis workspace's evidence section was renamed from "Evidence citations" to "Evidence supplied to analysis" (`apps/web/.../analyses/[id]/page.tsx`) — the old name implied every listed record was a citation, which was already misleading before this pass (bounded evidence often includes records the model never actually references) and became actively wrong once uncited-but-supplied rows exist. ## 2026-07-21 (Phase 4 correction pass) — Immutable, analysis-time readiness snapshot; historical briefings no longer recalculate current readiness

Confirmed as a real defect: the readiness briefing called `calculateReadinessScore(PROGRAM_ID)` live, on every render — so a historical analysis's displayed "program readiness" figure silently changed after later, unrelated program mutations (a risk closed, a milestone completed), even though every other figure on that same briefing (schedule exposure, budget exposure, the mitigation options themselves) was correctly frozen at analysis time. This directly contradicts the briefing's own purpose as a record of what was true when the analysis ran.
Fix: `ImpactAnalysis` gained `readinessSnapshot Json?` (same migration as above). `readinessSnapshotSchema` (`packages/core/src/ai/model-input.ts`) is the authoritative shape — reused, not reinvented: it's the same non-nullable object schema `ModelInputProjection.deterministicResults.readinessScore` already used (`{ totalScore, factors: [{ label, score, detail }] }`), now also exported and used as the persisted-snapshot content schema. The pure `buildSucceededImpactAnalysisData(output, modelInput)` (`attempt-persistence.ts`) reads `readinessSnapshot` directly from `modelInput.deterministicResults.readinessScore` — never from `output` (readiness was never part of the model's output schema to begin with — it has no model-supplied copy) and never recalculated. Persisted via `Prisma.DbNull` (not `Prisma.JsonNull`) when unavailable, since the intent is a real SQL `NULL` in this nullable `Json` column, not the stored JSON literal `"null"`.
The briefing page's `calculateReadinessScore(PROGRAM_ID)` call and its `PROGRAM_ID`/`calculateReadinessScore` imports were removed entirely — it now only ever reads `succeeded.readinessSnapshot`, the persisted value belonging to that specific successful attempt. The analysis workspace gained a new "Program readiness at analysis time" section reading the same persisted field (previously the workspace displayed no readiness figure at all).
Verified directly with an integration test (`readiness-snapshot.test.ts`, "[historical immutability]"): run a real mock analysis, capture its persisted snapshot, close an `OPEN` risk (confirmed via `calculateReadinessScore()` that this actually changes the current program-wide score — the test would be vacuous otherwise), reload the analysis, assert its stored snapshot is byte-identical to before, then restore the risk's original status. A separate test confirms the seeded demonstration analysis's persisted snapshot matches what the production pipeline independently computes for its event. "Readiness genuinely unavailable → persist null" is covered at the pure `buildSucceededImpactAnalysisData()` level (a fabricated `modelInput` with `readinessScore: null`) rather than via a real database scenario: `calculateReadinessScore()` only returns `{ok:false}` for a program ID that doesn't exist, which can't occur in `runImpactAnalysis()`'s real flow (the program ID always comes from an already-fetched, existing `ProgramEvent`'s own `program.id`), so there is no genuine database-backed way to exercise this path — pure-level coverage is the correct and complete test for it.
Alternatives: recompute readiness on read but cache it with a timestamp comparison against the analysis's `createdAt` (rejected — needlessly complex for what's fundamentally a "freeze this value at write time" requirement; a snapshot column is simpler and matches how `scheduleExposureDays`/`budgetExposureAmount` already work); store the readiness snapshot only on the briefing's own render path rather than on `ImpactAnalysis` (rejected — the analysis workspace needs the same historical value, and a snapshot that only exists conditionally on which page happened to render first isn't a real snapshot).
